## Purpose / Big Picture

After this change, a developer can run a single command such as:

    python -m backendapp.notebooklm.cli --files report.pdf slide.png --notebook-name "My Audio" --output-dir ./audio_out

and receive a downloaded audio file (MP3/WAV) generated by Google NotebookLM's Audio Overview feature — without writing any browser automation code themselves. The tool launches a Chromium browser (via Playwright), logs into NotebookLM using an existing Google session stored in a browser profile, creates a notebook, uploads the provided files, triggers audio generation, waits for completion, and downloads the result.

This matters because NotebookLM has no public REST API. Browser automation is the only viable path to programmatic audio generation from documents.

## Decision Log

- Decision: Use `playwright` (async API) with a persistent browser context (user-data-dir) rather than launching a fresh browser each run.
  Rationale: Google account authentication via OAuth is complex to automate. Reusing an existing browser profile that already has a valid Google session avoids the auth problem entirely, as stated in the requirements.
  Date/Author: 2026-03-22 / overnight-planner

- Decision: Structure code into four focused modules (`browser.py`, `uploader.py`, `audio.py`, `cli.py`) plus an `__init__.py`, as specified in the issue's directory layout.
  Rationale: Separation of concerns makes each piece independently testable and replaceable.
  Date/Author: 2026-03-22 / overnight-planner

- Decision: Use `asyncio` throughout; `playwright.async_api` requires it.
  Rationale: `playwright` Python bindings are async-first; mixing sync and async creates complexity.
  Date/Author: 2026-03-22 / overnight-planner

- Decision: Add `playwright` and `playwright-stealth` to `backendapp/requirements.txt`.
  Rationale: These are the only new runtime dependencies. `playwright-stealth` reduces the chance of bot-detection on notebooklm.google.com.
  Date/Author: 2026-03-22 / overnight-planner

## Context and Orientation

The repository lives at `/workspaces/dev-openclaw`. The existing Python backend is at `backendapp/` and has its own `requirements.txt` at `backendapp/requirements.txt`. There is no `notebooklm` sub-package yet; all files listed below must be created.

Key terms:

- **Playwright**: A Python library that drives a real browser (Chromium, Firefox, or WebKit) programmatically. It can click buttons, fill forms, wait for network responses, and intercept downloads. Installed with `pip install playwright` followed by `playwright install chromium`.
- **Persistent browser context** (`user_data_dir`): A Playwright feature that stores cookies, localStorage, and cached credentials in a folder on disk across runs. By pointing this at a Chrome/Chromium profile where the user has already signed into Google, the tool inherits the active session without needing to re-authenticate.
- **Audio Overview**: A NotebookLM UI feature (a panel accessible from the notebook page) that synthesises a spoken podcast-style summary of the uploaded sources. The user clicks "Generate" and NotebookLM returns a playable audio track.

### Step 3 — Implement `browser.py`

This module owns browser lifecycle. It exposes one async context-manager function `get_browser_page(user_data_dir: str | None, headless: bool) -> AsyncIterator[Page]` that:

1. Calls `async_playwright().start()`.
2. Launches a persistent Chromium context using `browser_type.launch_persistent_context(user_data_dir=..., headless=headless, ...)`. If `user_data_dir` is `None`, defaults to `~/.notebooklm_profile` so the first run creates the profile and subsequent runs reuse it.
3. Applies `playwright_stealth` to the first page to reduce bot-detection.
4. Yields the page.
5. On exit, calls `context.close()` and `playwright.stop()`.

Signature to define in `backendapp/notebooklm/browser.py`:

    from contextlib import asynccontextmanager
    from collections.abc import AsyncIterator
    from playwright.async_api import Page

    @asynccontextmanager
    async def get_browser_page(
        user_data_dir: str | None = None,
        headless: bool = True,
    ) -> AsyncIterator[Page]: ...

### Step 4 — Implement `uploader.py`

This module handles navigation to NotebookLM, notebook creation, and file upload.

Define one async function:

    async def create_notebook_and_upload(
        page: Page,
        notebook_name: str,
        file_paths: list[str],
    ) -> str:
        """Create a new notebook, upload files, return the notebook URL."""

Sequence of Playwright operations (use `page.wait_for_selector` / `page.locator(...).click()` / `expect` patterns):

1. Navigate to `https://notebooklm.google.com/`.
2. Wait for the page to be authenticated (presence of the "New notebook" button; if not found within 30 s, raise `RuntimeError("Not authenticated — open NotebookLM in the profile browser first")`).
3. Click "New notebook" (locator: button or link whose text matches "New notebook").
4. Wait for the notebook creation dialog; fill in the notebook name if a name input is present, then confirm.
5. Wait for the notebook editor page to load (URL should contain `/notebook/`).
6. Click the "Add source" or "+" source button.
7. In the source-addition dialog, choose "Upload file".
8. Use `page.set_input_files(selector, file_paths)` to supply all files at once (NotebookLM accepts multiple file uploads in a single dialog).
9. Confirm/submit the upload dialog.
10. Wait for all uploaded sources to appear as cards in the source panel (poll until the count of source cards equals `len(file_paths)` or a 5-minute timeout).
11. Return `page.url` as the notebook URL.

CSS selectors and aria labels for NotebookLM are subject to change; use `page.get_by_role` and `page.get_by_text` Playwright locators where possible because they are more resilient than raw CSS selectors. Add `# FRAGILE:` comments wherever raw selectors are used so a future maintainer knows to verify them.

### Step 5 — Implement `audio.py`

This module triggers Audio Overview generation and downloads the result.

Define one async function:

    async def generate_and_download_audio(
        page: Page,
        output_dir: str,
    ) -> str:
        """Trigger Audio Overview generation and return path to downloaded file."""

Sequence of operations:

1. Locate the "Audio Overview" panel. In NotebookLM's UI this is typically a collapsible section in the right-hand sidebar. Click its header to expand it.
2. Check whether audio has already been generated (presence of a play button / audio player element). If it has, skip to step 5.
3. Click the "Generate" button inside the Audio Overview panel.
4. Poll (every 10 seconds, up to 30 minutes) until the audio player element appears, indicating generation is complete. Log progress to stderr each poll cycle.
5. Once the player is visible, set up a download handler with `page.expect_download()` and click the download icon/button in the audio player.
6. Save the download to `output_dir` using `download.save_as(destination_path)`. The filename should be derived from `download.suggested_filename` if available, falling back to `audio_overview.mp3`.
7. Return the absolute path of the saved file.

### Step 6 — Implement `cli.py`

This is the CLI entry point, invoked with `python -m backendapp.notebooklm.cli`.

Use `argparse` (stdlib; no new dependencies). Define a `main()` function and guard it with `if __name__ == "__main__": asyncio.run(main())`. The module-level `__main__` block is also needed because of the `-m` invocation.

Arguments:

- `--files FILE [FILE ...]` (required): one or more paths to PDF or image files to upload.
- `--notebook-name TEXT` (optional, default `"Audio Overview <ISO timestamp>"`): name for the new notebook.
- `--output-dir DIR` (optional, default `"./notebooklm_audio"`): directory where the downloaded audio file is saved. Create it if it does not exist (`Path(output_dir).mkdir(parents=True, exist_ok=True)`).
- `--user-data-dir DIR` (optional, default `None` → `browser.py` picks `~/.notebooklm_profile`): path to the Chromium user profile directory.
- `--no-headless` (flag, default omitted → headless=True): show the browser window (useful for debugging and initial login).

The `main()` coroutine should:

1. Parse arguments.
2. Validate that every path in `--files` exists and has a `.pdf`, `.png`, or `.jpg`/`.jpeg` extension; exit with an error message if not.
3. Call `get_browser_page(...)` to obtain a `Page`.
4. Call `create_notebook_and_upload(page, notebook_name, file_paths)`.
5. Call `generate_and_download_audio(page, output_dir)`.
6. Print the path of the downloaded file to stdout.
7. Wrap all steps in `try/except Exception as e: print(f"Error: {e}", file=sys.stderr); sys.exit(1)`.

Full function signature skeleton:

    from datetime import datetime, timezone
    from pathlib import Path
    from backendapp.notebooklm.browser import get_browser_page
    from backendapp.notebooklm.uploader import create_notebook_and_upload
    from backendapp.notebooklm.audio import generate_and_download_audio

    def build_parser() -> argparse.ArgumentParser: ...

    async def main() -> None: ...

    if __name__ == "__main__":
        asyncio.run(main())

## Validation and Acceptance

The implementation is complete when all of the following are true:

1. `python -m backendapp.notebooklm.cli --help` exits 0 and prints usage with all documented options.
2. Running the tool with a valid PDF produces a playable MP3 or WAV file in the output directory.
3. Running the tool with both a PDF and a PNG uploads both files (verified by inspecting the notebook's source panel in the browser during a `--no-headless` run).
4. Running the tool with an invalid file path (e.g. `--files /nonexistent.pdf`) prints an error message to stderr and exits non-zero.
5. `ruff check backendapp/notebooklm/` exits 0.
6. `ruff format --check backendapp/notebooklm/` exits 0.


## Idempotence and Recovery

Each run creates a new notebook on NotebookLM (to avoid interference between runs). If a run fails partway through, it leaves behind an empty or partially populated notebook on NotebookLM, but no local side-effects beyond any partially written files in the output directory. Re-running is safe: a new notebook will be created.

If Playwright selectors break due to a NotebookLM UI update, look for `# FRAGILE:` comments in `browser.py`, `uploader.py`, and `audio.py` — these mark the places most likely to need updating.
